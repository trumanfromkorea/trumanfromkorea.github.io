## [코딩테스트] 프로그래머스 데브코스 2기 - 백엔드 엔지니어링

방금 막 프로그래머스 백엔드 데브코스 코딩테스트를 치뤘습니다. 시간은 4시간으로 진행되었고 문제는 총 25문제로 구성되어 있었습니다. 구글링도 모두 허용되었고 화면 녹화나 웹캠도 켜지 않은 채 상당히 널널한 환경에서 테스트가 진행되었습니다.

알고리즘 3문제, SQL 2문제, 객관식 20 문제로 구성되어 있었고, 알고리즘에 가장 많은 시간을 소요하게 될 것 같아 객관식 -> SQL -> 알고리즘 순으로 문제를 풀었습니다.

처음 풀기 시작할때는 시간이 진짜 넉넉하게 느껴졌는데 역시 막판에 다다르니 1시간만 더 있었더라면... 하는 생각이 들었습니다. 오랜만에 치룬 코딩테스트 치고는 나쁘지 않게 본거 같기는 한데, 결과가 나와보기 전에는 모르니 열심히 다른 공부도 소홀히 하지는 말아야겠다는 생각이 드네요.

## 문제 난이도

처음으로 풀었던 객관식 문제들은 그렇게 어렵지 않았습니다. IT 전공자 분들께서는 무난하게 통과하셨을 것 같습니다. 

저는 SQL 한지 오래되어서 그런가 좀 버벅였던 것 같습니다. 그래도 꾸준히 SQL 해오셨거나 잘 하시는 분들은 무난하게 정답 맞추셨을거라고 예상합니다.

## 알고리즘

코딩테스트를 준비한지 얼마 안되서 알고리즘 문제에 좀 두려움이 있긴 했는데, 그래도 옛날보다는 실력이 나아졌겠지 라는 생각에 자신감이 조금 붙었던 것 같습니다.

### 1번 문제

1번 문제는 탐색 문제였습니다. 문제 자체를 이해하는데는 그렇게 많은 시간이 걸리지 않았으나 문제를 어떻게 풀어나가느냐가 중요했던 문제가 아닌가 싶습니다.

맨 처음에는 DP 를 사용할지, 분할정복법을 이용할지, 이진탐색을 이용할지 등등 고민을 좀 했는데, 직접 손으로 적어가면서 풀어보니까 이것도 아닌것 같고 저것도 아닌것 같아서 시간을 많이 소요했습니다. 

결국 완전탐색으로 O(n^2) 의 시간복잡도를 가지는 방식으로 해결했는데, 잘하시는 분들께서는 더 효율적인 방법으로 해결하셨을것 같아 조금 궁금하네요...

### 2번 문제

2번 문제는 스택을 사용하는 문제였는데, 문제 자체도 너무 길고 조건도 너무 까다로워서 애를 먹었습니다. 문제의 난이도 자체는 그렇게 높지 않았지만 분기를 위해서 if/else 문을 사용하는 것이 정말 힘들었던 것 같습니다.

외에는 스택의 기본 기능인 push 와 pop 그리고 peek 을 이용해서 해결했습니다. 난이도 자체는 적당해서 생각나는대로 코드를 작성하기에 딱 좋았던 문제였습니다.

### 3번 문제

코딩테스트 볼 때마다 나오는 것 같은 그래프 탐색 문제... 예전에는 코딩테스트 볼 때 쳐다도 못봤던 문제였겠지만 이번에는 뭔가 풀 수 있을것만 같은 자신감에 상당히 노력했던 것 같습니다.

단순하게 DFS/BFS 를 사용하는 것이 아니라 가중치가 있기 때문에 다익스트라 알고리즘을 사용해야 하나 싶었는데, 사실 정확한 풀이법은 저도 아직 모르겠습니다.

저는 다익스트라 알고리즘을 사용하려고 했지만 응용이 필요한 문제였기 때문에 문제를 완벽히 해결하지는 못했고, 25개의 테스트 케이스 중 19개만을 통과하였습니다. 나머지 6개의 테스트 케이스는 메모리 초과로 실패했는데 테스트 시간이 5분 남은 시점이라 해결하지는 못하였습니다ㅜㅜ

아직 그래프 순회 알고리즘 공부를 제대로 안해서 그런지 구글링을 통해서 많은 도움을 얻었는데, 그래도 아예 못풀던 문제를 건드려라도 봤으니 조금은 성장하지 않았나 싶기도 합니다.

## 마무리

사실 자소서도 그렇게 잘 쓴것 같지도 않고 코딩테스트도 좋은 성적을 거두지는 못한 것 같아서 큰 기대는 안하지만, 면접 보게 된다면 면접 준비도 열심히 해야겠죠?
